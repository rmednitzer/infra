# ADR-0006: Code audit 2026-05 findings

- **Status**: Accepted
- **Date**: 2026-05-24

## Context

This ADR records the findings of a full repository audit performed on
2026-05-24, in which every `.tf`, `.cfg`, `.yml`, `.hcl`, `.sh`, and `.md`
file in the repository was re-read and cross-checked against the current
upstream documentation for:

- OpenTofu (latest: 1.12.0)
- `dmacvicar/libvirt` provider (latest in pin range: 0.8.3; latest overall:
  0.9.7)
- The cloud-init NoCloud datasource
- The S3 backend documentation for OpenTofu 1.10+
- The GitHub Actions used in CI (`actions/checkout`, `opentofu/setup-opentofu`,
  `terraform-linters/setup-tflint`)

The audit was the second of its kind; the first (PR #7) hardened the
`libvirt-vm` module, enforced TFLint in CI, and reconciled doc/code drift.
This ADR is intended to be the durable record of what *this* round of the
audit found, what we changed, and what we deferred.

## Decision

For each finding below, we record severity, the upstream evidence, and the
chosen remediation.

### Finding 1 — Production `backend.tf` example used deprecated S3 syntax (FIXED)

**Severity**: Medium (documentation-only; affected the commented example,
not any live backend).

**Evidence**: OpenTofu 1.10 deprecated three top-level S3 backend attributes
in favour of newer alternatives:

| Deprecated | Replacement |
|------------|-------------|
| `endpoint = "…"` | `endpoints = { s3 = "…" }` |
| `force_path_style = true` | `use_path_style = true` |
| `dynamodb_table = "…"` | `use_lockfile = true` (native S3 conditional writes) |

The commented example in `environments/production/backend.tf` used all three
deprecated forms. Anyone copy-pasting it as a starting point would inherit
the deprecations.

**Resolution**: Rewrote the commented example to use `endpoints`,
`use_path_style`, and `use_lockfile`. Added a comment noting that
`dynamodb_table` is still accepted by OpenTofu for backward compatibility,
but new deployments should prefer `use_lockfile`. See ADR-0003.

### Finding 2 — `libvirt_cloudinit_disk` does not set `meta_data` (DEFERRED)

**Severity**: Low (the resource works in practice with modern Ubuntu cloud
images, but the behaviour is implicit).

**Evidence**: The cloud-init NoCloud datasource documentation states:
"meta-data … is required to contain an `instance-id`." Inspection of the
`dmacvicar/libvirt` provider source (`v0.8.3/libvirt/cloudinit_def.go`)
shows the provider does not generate a default `instance-id` when
`meta_data` is empty — it writes an empty file. Modern Ubuntu cloud-init
tolerates this via first-boot heuristics, but the contract is not
guaranteed and the instance-id will not be deterministic across re-creates.

**Resolution**: Documented in ADR-0004. Recommended follow-up is to set
`meta_data = "instance-id: ${var.vm_name}\nlocal-hostname: ${var.vm_name}\n"`
on the `libvirt_cloudinit_disk` resource. This change is **deferred** to a
separate PR because it forces a `libvirt_cloudinit_disk` re-creation on
next apply for any operator with existing infrastructure, which triggers
a domain restart.

### Finding 3 — Libvirt provider 0.9.x is available but is a breaking redesign (NO CHANGE)

**Severity**: Informational.

**Evidence**: `dmacvicar/libvirt` 0.9.0 (Nov 2025) introduced a major
redesign on the new Terraform plugin framework with full libvirt API
coverage. Seven point releases have shipped in the 0.9.x line since then.
The 0.8.x line continues to receive patch fixes (0.8.3 in Mar 2026).

**Resolution**: No change. The current pin `~> 0.8.0` is correct and
intentional; the rationale and a migration plan are now documented in
ADR-0002.

### Finding 4 — OpenTofu version pin is broader than necessary for the production backend (DOCUMENTED)

**Severity**: Low.

**Evidence**: `required_version = ">= 1.6"` in every `versions.tf`. The
production backend, per ADR-0003, will use `use_lockfile`, which requires
OpenTofu 1.10 or newer. The current pin lets a 1.6-era client try to apply
production and fail at runtime rather than at `tofu init`.

**Resolution**: Recorded in ADR-0003 as a follow-up. When
`environments/production/backend.tf` is switched from the local placeholder
to the real S3 backend, bump that environment's `required_version` to
`>= 1.10` in the same PR. The lab environment can stay at `>= 1.6`.

### Finding 5 — CI Action versions are current (NO CHANGE)

**Severity**: Informational.

**Evidence**:

| Action | Pinned | Upstream latest | Status |
|--------|--------|-----------------|--------|
| `actions/checkout` | v6 | v6.0.2 (Jan 2025) | Current |
| `opentofu/setup-opentofu` | v2 | v2.0.1 (May 2024) | Current |
| `terraform-linters/setup-tflint` | v6 | v6.2.2 (Mar 2024) | Current |

Dependabot is configured (`.github/dependabot.yml`) to keep these in
sync going forward.

**Resolution**: No change.

### Finding 6 — `tflint.hcl` uses modern syntax (NO CHANGE)

**Severity**: Informational.

**Evidence**: `.tflint.hcl` uses `call_module_type = "all"`, which is the
TFLint 0.50+ replacement for the older `module = true` setting. Both forms
work in current TFLint, but the new one is the documented spelling.

**Resolution**: No change.

### Finding 7 — Module assumes `default` libvirt pool and network exist (DOCUMENTED)

**Severity**: Low.

**Evidence**: `libvirt-vm` defaults `storage_pool = "default"` and
`network_name = "default"`. On a freshly-installed libvirt host these
typically exist, but neither is guaranteed and the module does **not**
create them. Applying against a host without those resources produces a
provider error at apply time.

**Resolution**: Updated `modules/libvirt-vm/README.md` to call out the
assumption explicitly under "Requirements" and "Notes". Creating the pool
and network is out of scope for this module — they are host-level concerns
and would belong in a separate module if we ever automate host setup.

### Finding 8 — Documentation cross-references are missing (FIXED)

**Severity**: Low.

**Evidence**: `README.md`, `CLAUDE.md`, `modules/libvirt-vm/README.md`, and
`.github/copilot-instructions.md` each carry conventions and rationale.
Until now, none of them pointed at a durable decision record.

**Resolution**: Created `docs/adr/` and added cross-references from each
top-level doc to the relevant ADRs. See updated files in this PR.

## Consequences

- Future readers (human or AI) have a single place — `docs/adr/` — to look
  up *why* a choice was made, not just *what* the choice is.
- The deferred items (Finding 2, Finding 4) are now scheduled work, not
  oral tradition.
- The repository's compliance posture (state encryption, locking,
  hardened cloud-init defaults) is documented in `ADR-0003` and `ADR-0004`
  rather than scattered across README sections.
